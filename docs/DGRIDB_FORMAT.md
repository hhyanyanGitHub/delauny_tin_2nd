# DGRIDB 1 二进制 GRID 格式

DGRIDB 面向本机大 GRID 的快速持久化、写时复制映射和屏幕概览。它不是跨端序交换
格式；需要长期人工审阅或异构系统交换时仍可使用 DGRID 文本、GeoTIFF/COG。

## 文件布局

所有整数为无符号小端序，浮点数为 IEEE-754 little-endian `double`。v0.29 文件由六段组成：

1. 固定 65,536 字节头；
2. 最多 512×512 个 `double` 的兼容全幅平均概览；
3. 金字塔目录；
4. 各个按 65,536 字节边界对齐的 2× 平均金字塔层；
5. 原始节点 4 MiB 分块校验表；
6. `width * height` 个按行优先排列的原始 `double` 节点。

加载器校验魔数、版本、端序、头校验、尺寸乘法、段偏移、统计计数和实际文件大小。
头校验覆盖扩展指针，但不覆盖概览、目录和数据段。`dt_grid_verify_binary_file()` 会扫描
并核对全部原始节点块；若还需要覆盖金字塔或抵抗恶意篡改，应在文件外增加 SHA-256
等密码学校验。

每条金字塔目录记录固定 32 字节：偏移 0 为 32 位缩放倍数，偏移 4 为零保留字段，
偏移 8/16 为 64 位宽高，偏移 24 为 64 位数据偏移。层级从 scale=2 开始逐次加倍，
宽高分别为 `ceil(width/scale)` 和 `ceil(height/scale)`；最后一级最长边不超过 512。
每层数据仍是按行优先的 double。

扩展保持旧读兼容：兼容全幅概览仍位于 65,536，原始节点实际位置仍由旧字段 128
给出；v0.28 加载器会忽略 200～255 的扩展字段和中间新增段，直接映射原始节点。
v0.29 加载器在扩展魔数缺失时按 v0.28 布局处理。

## 映射与编辑语义

Windows 加载器以 `PAGE_WRITECOPY` / `FILE_MAP_COPY` 分别映射原始节点和各金字塔层。
读取只在访问相应页面时由系统调页；`dt_grid_write_window()` 修改私有页，不会改变源
DGRIDB。所有现有 GRID 查询、转换和分析函数继续接收同一个 `dt_grid_handle`。

`dt_grid_info.flags` 的 `DT_GRID_STORAGE_MEMORY_MAPPED` 表示当前节点仍由映射支持，
`DT_GRID_HAS_PERSISTENT_OVERVIEW` 表示可直接复用内置全幅平均概览，
`DT_GRID_HAS_PYRAMID` 和 `DT_GRID_HAS_BLOCK_CHECKSUMS` 表示扩展能力。这些标志只用于
输出，不应传给 `dt_grid_create()`。`dt_grid_prefetch_window()` 在 Windows 上用
`PrefetchVirtualMemory` 提示即将读取的原始映射范围；不可用时安全退化为空操作。

显式调用 `dt_grid_save_binary()` 才提交数据。保存先在目标目录完整写临时文件并刷新，
成功后原子替换目标；失败时已有目标不变。若保存目标就是当前映射源，库在临时文件
完成后将当前私有视图实体化并解除映射，再执行替换，因此该特殊路径会短时需要一份
完整节点内存。另存为其他文件不需要解除源映射。

## 持久概览

保存时用 `DT_GRID_OVERVIEW_AVERAGE` 为完整源范围生成最多 512×512 的概览，并记录
完整源统计。加载后，以相同尺寸、平均方法、默认 NoData 策略请求完整源范围时，
`dt_grid_read_overview()` 直接复制该概览；局部视口或其他方法仍读取原始映射节点。
任意节点写入会使旧概览失效；下一次二进制保存会重新生成并嵌入新概览。

## 多级金字塔与块校验

保存从原始层开始逐级对 2×2 节点求有效值平均，NoData 不参与平均；四个节点均无效时
输出 NoData。`DT_GRID_OVERVIEW_USE_PYRAMID` 仅为平均、非严格 NoData 的显示概览选择
仍不小于目标输出的最低分辨率层。使用后结果设置 `DT_GRID_OVERVIEW_USED_PYRAMID`，且
不会设置精确源统计位。其他方法和未显式启用的调用仍读取原始节点。

校验表按原始 double 字节流每 4 MiB 一项，最后一块可以较短。普通加载只检查表的布局，
不扫描原始数据，因此不会牺牲快速打开；`dt_grid_verify_binary_file()` 才执行完整扫描。
写入任意节点后，当前句柄的金字塔和块校验能力位会清除，避免继续使用陈旧派生数据；
下一次 `dt_grid_save_binary()` 会在临时文件中重建两者后再原子替换。
